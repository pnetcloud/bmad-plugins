# Framework Notes

Use this file when the requested implementation depends on framework-specific rendering behavior.

## Goal

Translate the general rendering and metadata rules into practical patterns for common web stacks.

## Next.js

### App Router

- use route-level metadata APIs instead of client-side tag mutation
- generate canonicals and alternates deterministically
- ensure server components or route handlers provide the primary rendered content
- watch ISR and revalidation so stale metadata or schema do not linger

### Pages Router Or Mixed Apps

- keep metadata generation centralized and predictable
- verify that older client-side patterns are not overriding server intent

## Astro

- pre-render indexable pages whenever feasible
- avoid client islands for the only meaningful content
- if dynamic data is required, ensure the page still returns the critical copy in HTML

## Nuxt

- use head management and route metadata in a deterministic way
- avoid client-only wrappers for ranking-critical sections
- confirm async data required for the primary content resolves into server output

## SPA-Heavy Apps

- if a page should rank, provide SSR, SSG, or a server-rendered fallback for core content
- do not assume a client-rendered app shell is enough for meaningful discovery
- define how canonical, schema, and body copy are produced before hydration

## Framework Migration Watchlist

Re-audit SEO after:

- metadata API migration
- route migration
- server/client boundary changes
- caching or revalidation changes
- component refactors that move copy into client-only UI

## Verification Checklist

- Is the primary content present in rendered HTML?
- Are title, meta description, canonical, and schema generated by a stable server-side path?
- Can the framework’s cache or revalidation layer serve stale SEO signals?
- Did the migration move important content behind hydration?
